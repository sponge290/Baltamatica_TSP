# 天气感知型 TSP 求解系统使用教程

## 1. 项目概述

天气感知型 TSP 求解系统是一个基于北太天元v3.6的路径规划工具，采用「公网静态前端 + 用户本地北太天元计算」的创新架构，实现了考虑天气因素对旅行时间影响的TSP问题求解。

### 核心功能
- 支持3种TSP求解算法：动态规划（DP）、A*、遗传算法（GA）
- 考虑天气因素对旅行时间的影响
- 支持时间窗口约束
- 公网前端托管于Vercel，提供美观的可视化界面
- 本地代理服务，驱动北太天元执行算法计算
- 结果持久化到Supabase数据库，支持分享和对比

### 技术架构
- **公网前端层**：Vercel静态托管 + HTML5/CSS3/JS + D3.js v7
- **数据持久层**：Supabase (PostgreSQL 15+)
- **本地代理层**：北太天元v3.6原生.m脚本（内嵌Python HTTP服务）
- **核心计算层**：北太天元v3.6

## 3. 部署步骤

### 3.1 数据库初始化（Supabase）

1. **创建Supabase项目**
   - 访问 [Supabase官网](https://supabase.com) 注册账号
   - 创建新项目，选择就近的区域
   - 记录项目URL和公钥（后续配置需要）

2. **执行数据库初始化脚本**
   - 登录Supabase控制台，进入SQL编辑器
   - 复制并执行 `project/scripts/supabase_init.sql` 文件内容
   - 等待执行完成，确认所有表结构创建成功

3. **配置环境变量**
   - 在项目根目录创建 `.env` 文件
   - 填写Supabase连接信息：
     ```
     VITE_SUPABASE_URL=你的项目URL
     VITE_SUPABASE_ANON_KEY=你的公钥
     ```

### 3.2 本地代理配置

1. **下载代理脚本**
   - 从项目中获取 `project/proxy/baltamatica_tsp_proxy.m` 文件
   - 在北太天元v3.6中打开该文件

2. **配置代理脚本**
   - 修改脚本中的 `config.allow_origin` 配置，添加你的Vercel域名和本地开发地址
   - 保持其他配置不变

3. **启动代理服务**
   - 在北太天元v3.6中运行脚本
   - 看到启动成功提示后，保持窗口开启

### 3.4 前端部署

1. **安装依赖**
   ```bash
   cd project/frontend
   npm install
   ```

2. **本地开发测试**
   ```bash
   npm run dev
   ```
   - 访问 `http://localhost:3000` 测试前端功能

3. **构建生产版本**
   ```bash
   npm run build
   ```

4. **部署到Vercel**
   - **步骤1：创建GitHub仓库**
     - 登录GitHub，创建一个新的仓库
     - 将项目代码推送到GitHub仓库
     - 确保包含所有必要文件，包括 `package.json`、`vite.config.js` 等

   - **步骤2：登录Vercel**
     - 访问 [Vercel官网](https://vercel.com) 并登录
     - 点击「Add New」按钮，选择「Project」

   - **步骤3：导入项目**
     - 选择「Import from GitHub」
     - 搜索并选择你的GitHub仓库
     - 点击「Import」按钮

   - **步骤4：配置项目**
     - 项目名称：可使用默认名称或自定义
     - 框架预设：选择「Vite」
     - 根目录：保持默认（或设置为 `project/frontend`）
     - 构建命令：保持默认 `npm run build`
     - 输出目录：保持默认 `dist`

   - **步骤5：配置环境变量**
     - 点击「Environment Variables」部分
     - 添加以下环境变量：
       - `VITE_SUPABASE_URL`：你的Supabase项目URL
       - `VITE_SUPABASE_ANON_KEY`：你的Supabase匿名密钥
     - 点击「Add」按钮保存

   - **步骤6：部署项目**
     - 点击「Deploy」按钮开始部署
     - 等待部署完成，Vercel会生成一个默认域名（如 `your-project.vercel.app`）

   - **步骤7：更新本地代理配置**
     - 复制Vercel生成的域名
     - 编辑 `baltamatica_tsp_proxy.m` 文件
     - 在 `config.allow_origin` 中添加你的Vercel域名
     - 重启本地代理服务

   - **步骤8：验证部署**
     - 访问Vercel生成的域名
     - 点击「检测」按钮，确认本地代理连接成功
     - 测试算法求解功能，确保系统正常运行

### 3.5 数据导入

1. **安装Python依赖**
   ```bash
   pip install pandas supabase-py
   ```

2. **执行数据导入**
   ```bash
   python project/scripts/data_importer.py --url 你的Supabase URL --key 你的服务密钥 --cities project/data/cities.csv --segments project/data/road_segments.csv --test-cases project/data/test_cases.csv
   ```

## 4. 使用流程

### 4.1 启动系统

1. **启动本地代理**：在北太天元v3.6中运行代理脚本
2. **访问前端网站**：打开部署在Vercel的前端网站
3. **连接检测**：点击导航栏的「检测」按钮，确认本地代理连接成功

### 4.2 算法求解

1. **选择测试用例**：从下拉菜单选择预设测试用例
2. **配置算法**：勾选需要执行的算法，调整遗传算法参数
3. **开始计算**：点击「开始计算」按钮
4. **查看结果**：计算完成后，查看可视化结果和性能指标
5. **保存结果**：点击「保存结果」按钮，将结果上传到Supabase

### 4.3 结果管理

1. **历史记录**：在「历史记录」页面查看所有保存的求解结果
2. **结果对比**：在「结果对比」页面比较不同算法的性能
3. **分享结果**：点击分享按钮，生成分享链接

## 5. 算法选择指南

| 算法 | 适用规模 | 特点 | 推荐场景 |
|------|----------|------|----------|
| 动态规划(DP) | ≤15个城市 | 全局最优解，计算时间随城市数量指数增长 | 小规模问题验证 |
| A*算法 | 10-30个城市 | 兼顾速度和最优性，使用启发式函数剪枝 | 中等规模问题 |
| 遗传算法(GA) | 30-100个城市 | 近似最优解，适合大规模问题 | 大规模业务场景 |

## 6. 常见问题与故障排除

### 6.1 本地代理连接失败
- **问题**：前端显示「未连接」状态
- **排查**：
  1. 检查北太天元是否正常运行
  2. 确认代理脚本是否已执行
  3. 检查端口18080是否被占用
  4. 确认前端域名已添加到代理脚本的 `allow_origin` 配置

### 6.2 算法执行超时
- **问题**：计算过程中出现超时错误
- **解决方案**：
  1. 对于大规模问题，使用遗传算法
  2. 调整遗传算法参数（减少种群规模或迭代次数）
  3. 增加代理脚本中的 `max_exec_time` 配置

### 6.3 结果上传失败
- **问题**：保存结果时出现错误
- **排查**：
  1. 检查Supabase环境变量是否正确配置
  2. 确认RLS安全策略是否正确设置
  3. 检查网络连接是否正常

### 6.4 可视化不显示
- **问题**：前端可视化面板无内容
- **排查**：
  1. 检查浏览器控制台是否有JavaScript错误
  2. 确认算法返回的结果格式是否正确
  3. 检查D3.js版本是否兼容

## 7. 性能优化建议

### 7.1 前端优化
- 使用现代浏览器，确保启用硬件加速
- 对于大规模问题，使用遗传算法以获得更快的响应时间
- 关闭不必要的浏览器扩展

### 7.2 算法参数优化
- **遗传算法**：
  - 种群规模：50-100（平衡精度和速度）
  - 迭代次数：100-300（根据问题规模调整）
  - 变异率：0.05-0.2（避免过早收敛）

### 7.3 系统资源管理
- 北太天元运行时，确保系统有足够的内存和CPU资源
- 对于大规模问题，建议在性能较好的计算机上运行
- 定期清理浏览器缓存，保持前端响应速度

## 8. 安全注意事项

### 8.1 本地代理安全
- 代理服务仅监听 `127.0.0.1`，禁止修改为 `0.0.0.0`
- 严格配置CORS跨域规则，仅允许项目前端域名访问
- 算法执行设置超时保护，防止资源耗尽

### 8.2 数据安全
- Supabase启用行级安全策略(RLS)，保护用户数据
- 敏感数据仅在本地计算，不上传到公网
- 定期备份Supabase数据库

### 8.3 授权合规
- 严格遵守北太天元授权协议，所有计算在本地执行
- 仅提供代理脚本下载，不打包分发北太天元核心文件
- 明确标注北太天元的知识产权归属
